# Code Review — Spread (Crypto Spread Monitor)

## 1. Архитектура и структура

**Плюсы:**
- Чёткое разделение: `app` (UI), `core` (модели, сервисы, хранилище), `exchange` (клиенты бирж), `tools` (CoinListGenerator).
- Использование интерфейса `ExchangeClient` и `ExchangeManager` упрощает добавление новых бирж.

**Рекомендации:**
- **Вынести модели UI из MainApp:** `TrackedCoin` и `ArbitrageRow` лучше перенести в отдельные классы (например, `com.spread.app.model`), чтобы уменьшить размер MainApp и упростить тестирование.
- **Константы путей:** пути `assets/coins.json`, `config/user-coins.json`, `config/settings.json`, `log/` захардкожены в нескольких местах. Имеет смысл вынести их в один класс конфигурации или в `Settings`.

---

## 2. MainApp (UI)

**Плюсы:**
- Вкладки разделены логично, привязки через `Bindings` для счётчика монет сделаны корректно.
- Обработка ошибок с диалогами и логированием на месте.

**Проблемы и улучшения:**

| Проблема | Рекомендация |
|----------|---------------|
| **Неиспользуемое поле `depositFieldRef`** | Удалить поле и присвоение — оно нигде не используется. |
| **Start при пустом списке монет** | При нажатии Start, если ни одна монета не включена или список пуст, вызывается `connectAll(empty)` и `startScheduler()`. Стоит проверять `symbols.isEmpty()` и показывать предупреждение или блокировать кнопку. |
| **Дублирование форматтеров таблицы арбитража** | Один и тот же код `TableCell` для цен/прибыли/спреда повторяется 4 раза. Вынести в приватные методы `createPriceCell(DecimalFormat)` и `createPercentCell(DecimalFormat)` или в общий хелпер. |
| **Deprecated `TableView.CONSTRAINED_RESIZE_POLICY`** | В JavaFX 20+ использовать `TableView.constrainedResizePolicy()` (фабричный метод). Для обратной совместимости можно оставить текущий вариант и закомментировать в ревью. |
| **Создание потоков через `new Thread()`** | Для фоновых задач (обновление списка монет, проверка при добавлении) лучше использовать общий `ExecutorService` (например, `Executors.newCachedThreadPool()` или отдельный пул для IO), чтобы ограничить число потоков и централизованно обрабатывать ошибки. |

## 3. Хранение данных и логирование

**CoinStorage / SettingsStorage:**
- При ошибке записи только `System.err`, данные могут не сохраниться — пользователь не узнает. Имеет смысл логировать в `AppLog.error` и по возможности показывать диалог при критичных ошибках (например, при сохранении настроек).
- **CoinStorage.addUserCoin:** при `path.getParent()` == null (корневой путь) `Files.createDirectories` может вести себя неочевидно. Для `config/user-coins.json` родитель есть, но проверка на null повысит надёжность.

**AppLog:**
- `format()` с `replaceFirst("\\{\\}", ...)` при нескольких плейсхолдерах работает корректно. Учтите: если в сообщении встречается `{}` не как плейсхолдер, его тоже заменит первый аргумент — для логов обычно приемлемо.
- Нет ротации логов — `app.log` и `kucoin-error.log` будут расти. Для продакшена стоит добавить простую ротацию по размеру или по дате (или перейти на SLF4J + Logback с file appender и ротацией).

## 4. Exchange и CoinListGenerator

**Exchange clients (Binance, Bybit, OKX, KuCoin):**
- Много дублирования: структура `connect`/`disconnect`, реконнект в `onFailure`, парсинг JSON. Можно вынести общую логику в абстрактный `BaseExchangeClient` (или делегировать реконнект одному компоненту).
- **KucoinClient:** запись ошибок в `log/kucoin-error.log` — хорошее решение для отладки. Остальные клиенты при падении только пишут в консоль; для единообразия можно логировать и их в `AppLog` или в отдельный `exchange-error.log`.

**CoinListGenerator:**
- **checkSymbolSupport** каждый раз запрашивает все 4 биржи заново — при частых проверках это тяжёло. Для кэширования можно хранить последний результат по символу с TTL (например, 5–10 минут), с осторожностью из-за делистинга.
- Дублирование блока try/catch по биржам в `generateAndSave()` можно сократить: список пар (название биржи, Supplier<Set<String>>), цикл с try/catch и общим учётом `okCount` и `incrementCounts`.

**ExchangeManager:**
- Один общий `OkHttpClient` для всех клиентов — правильно. При желании можно задать общий таймаут и пул соединений в одном месте.

## 5. Потоки и ресурсы

- **Scheduler:** `Executors.newSingleThreadScheduledExecutor()` создаётся при каждом Start, старый не переиспользуется после Stop — поведение корректное, т.к. при Stop делается `shutdownNow()`.
- **Platform.runLater:** используется по назначению для обновления UI из фоновых потоков.
- **WebSocket клиенты:** реконнект через общий `RECONNECT_SCHEDULER` — нормально; при массовом рестарте можно ограничить частоту реконнектов (backoff уже есть).
- **PriceAggregator:** при отключении бирж старые тикеры остаются в `tickersBySymbol`. При повторном Connect данные смешиваются. Имеет смысл при `disconnectAll()` очищать агрегатор или при `connect(symbols)` очищать тикеры по тем биржам, которые переподключаются, чтобы не показывать устаревшие цены.

---

## 6. Прочее

- **ArbitrageCalculator:** логика ясная; при большом числе символов и бирж можно кэшировать пары (buyEx, sellEx) и не создавать лишние объекты, но для 4 бирж это не критично.
- **Модели (Settings, Ticker, ArbitrageOpportunity):** неизменяемость где возможно соблюдена, структура понятная.
- **Тесты:** есть `ArbitrageCalculatorTest`, `PriceAggregatorTest` — хорошо. Стоит добавить тесты на `CoinStorage` (чтение/запись, merge, clearAll) и на форматирование в UI (опционально).

---

## 7. Приоритеты внедрения

1. **Высокий:** убрать неиспользуемое поле; проверка пустого списка при Start; единый ExecutorService для фоновых задач.
2. **Средний:** вынести форматтеры ячеек таблицы в методы; логирование ошибок записи в AppLog; очистка PriceAggregator при disconnect (или при connect).
3. **Низкий:** ротация логов; кэш для checkSymbolSupport; рефакторинг клиентов бирж в базовый класс; вынос TrackedCoin/ArbitrageRow в отдельные файлы.
